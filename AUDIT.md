# AUDIT.md — Инструкция по проверке проекта CI/CD воркшопа

## Задача

Проверь проект на работоспособность, корректность и качество учебных материалов.
Исправь все найденные проблемы. По итогу выдай отчёт.

---

## Шаг 1 — Установка и запуск

```bash
npm install
```

Убедись что установка завершается без ошибок и без критических предупреждений.

---

## Шаг 2 — Проверка всех скриптов

Последовательно выполни каждую команду и убедись что она завершается с кодом 0:

```bash
npm run type-check   # не должно быть ошибок TypeScript
npm run lint         # не должно быть ошибок ESLint
npm run test         # все тесты должны пройти
npm run build        # dist/ должен создаться без ошибок
```

Если что-то падает — исправь ошибку и перепроверь.

---

## Шаг 3 — Проверка тестов

Убедись что:

- Все тесты в `src/utils/filterTodos.test.ts` осмысленны и покрывают реальную логику
- Все тесты в `src/components/TodoItem.test.tsx` проверяют поведение компонента, а не детали реализации
- Тесты не завязаны на конкретные строки кода, которые могут измениться
- Названия тестов читаются как документация (`it('возвращает только активные задачи при фильтре active')`)

---

## Шаг 4 — Проверка workflow файлов

Для каждого из 6 файлов в `.github/workflows/` проверь:

**Синтаксис и корректность:**

- Валидный YAML (корректные отступы, нет лишних символов)
- Все `uses:` ссылаются на реально существующие actions с корректными версиями:
  - `actions/checkout@v5` ✓
  - `actions/setup-node@v5` ✓
  - `actions/upload-artifact@v4` ✓
  - `actions/download-artifact@v4` ✓
  - `peaceiris/actions-gh-pages@v5` ✓
  - `appleboy/ssh-action@v1` ✓
- Все `${{ secrets.NAME }}` соответствуют секретам, описанным в комментариях и README
- `needs:` ссылается на job'ы, которые реально существуют в том же файле

**Логика пайплайнов:**

- `ci-1`: только `checkout` + `setup-node` + `npm install` + `build`. Никаких лишних шагов — это намеренно простой пример
- `ci-2`: добавлены lint, type-check, test. Порядок шагов: lint → type-check → test → build (Fail Fast)
- `ci-3`: lint и test в отдельных параллельных jobs, build зависит от обоих через `needs`, артефакт загружается
- `cd-1`: использует `GITHUB_TOKEN` (не кастомный секрет), есть `permissions: contents: write`
- `cd-2`: использует три секрета (`SSH_PRIVATE_KEY`, `SERVER_HOST`, `SERVER_USER`), скрипт деплоя логичен
- `cd-3`: два deploy job'а с условиями `if:` по веткам, оба используют `download-artifact` (не пересобирают)

**Триггеры:**

- `ci-1` — единственный файл с автоматическим триггером `on: push`. Остальные должны иметь `on: workflow_dispatch` чтобы студент активировал их вручную

---

## Шаг 5 — Проверка учебных материалов

**Комментарии в workflow файлах:**

- Каждый новый концепт объяснён в комментарии при первом появлении
- Комментарии написаны на понятном языке, без избыточного жаргона
- Блок `ЗАДАНИЕ` в конце каждого файла содержит конкретные, выполнимые шаги
- Задания логически вытекают из того, что было показано в данной итерации

**Прогрессия сложности:**

- Убедись что каждая итерация добавляет ровно один новый слой концептов, не больше
- CI-1 → CI-2: новое — это PR триггер, npm ci, порядок шагов
- CI-2 → CI-3: новое — это параллельные jobs, needs, артефакт
- CD-1 → CD-2: новое — это реальные секреты, SSH
- CD-2 → CD-3: новое — это if-условия, environments, approval

**README:**

- Все команды в разделе "Быстрый старт" работают
- Описание каждой итерации соответствует тому, что реально находится в workflow файле
- Таблица пайплайнов в конце актуальна
